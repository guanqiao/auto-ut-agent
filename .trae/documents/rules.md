# PyUT Agent 代码整合 - 规则和约束

## 1. 代码迁移规则

### 1.1 废弃代码处理规则

#### RULE-001: 废弃代码保留原则
- **规则**: 所有标记为废弃的代码必须保留，不得删除
- **原因**: 保持向后兼容性
- **验证**: 确保旧代码仍可导入和运行
- **例外**: 明确标记为内部API且未在文档中公开的代码

#### RULE-002: 废弃警告规范
- **规则**: 所有废弃代码必须发出 `DeprecationWarning`
- **要求**:
  - 模块级废弃: 在模块顶部使用 `warnings.warn()`
  - 类/函数级废弃: 在 `__init__` 或函数开始处发出警告
  - 必须包含迁移建议: "Use X instead"
  - 必须设置 `stacklevel=2`
- **示例**:
```python
import warnings
warnings.warn(
    "OldClass is deprecated. Use NewClass instead. "
    "See: https://docs.pyutagent.org/migration",
    DeprecationWarning,
    stacklevel=2
)
```

#### RULE-003: 废弃代码冻结
- **规则**: 废弃代码进入冻结状态，不再接受功能更新
- **允许操作**:
  - Bug修复
  - 安全补丁
  - 文档更新
- **禁止操作**:
  - 新功能添加
  - API变更
  - 行为修改

### 1.2 迁移实施规则

#### RULE-004: 小步提交原则
- **规则**: 每次迁移只修改一个文件或一个功能点
- **提交信息格式**:
```
Migrate <file>: <old_interface> -> <new_interface>

- Replace <old_import> with <new_import>
- Update type annotations
- Maintain backward compatibility
```

#### RULE-005: 测试先行原则
- **规则**: 迁移前必须确保有充分的测试覆盖
- **要求**:
  - 单元测试覆盖率 > 80%
  - 集成测试覆盖主要使用场景
  - 迁移后所有测试必须通过

#### RULE-006: 功能等价验证
- **规则**: 迁移后必须验证功能等价
- **验证方法**:
  - 对比迁移前后的输出
  - 运行相同的测试用例
  - 检查边界条件处理

### 1.3 适配器规则

#### RULE-007: 适配器必须完整
- **规则**: 适配器必须实现原接口的所有公共方法
- **要求**:
  - 方法签名保持一致
  - 行为保持一致
  - 文档说明适配关系

#### RULE-008: 适配器性能
- **规则**: 适配器引入的性能开销必须 < 10%
- **验证**: 基准测试对比

## 2. 代码质量规则

### 2.1 代码风格规则

#### RULE-009: 遵循PEP 8
- **规则**: 所有代码必须符合 PEP 8 规范
- **工具**: 使用 `ruff` 或 `flake8` 检查
- **例外**: 已有代码保持原风格，新代码必须遵循

#### RULE-010: 类型注解
- **规则**: 所有公共API必须有类型注解
- **要求**:
  - 函数参数类型
  - 返回值类型
  - 类属性类型
- **工具**: 使用 `mypy` 检查

#### RULE-011: 文档字符串
- **规则**: 所有公共类和方法必须有文档字符串
- **格式**: Google Style 或 NumPy Style
- **要求**:
  - 描述功能
  - 参数说明
  - 返回值说明
  - 异常说明

### 2.2 代码结构规则

#### RULE-012: 单一职责原则
- **规则**: 每个类/函数只负责一个功能
- **限制**:
  - 类行数 < 300 行
  - 函数行数 < 50 行
  - 参数数量 < 5 个

#### RULE-013: 导入规范
- **规则**: 导入必须按顺序组织
- **顺序**:
  1. 标准库
  2. 第三方库
  3. 本地模块（绝对导入）
  4. 本地模块（相对导入）
- **格式**:
```python
# 标准库
import os
from typing import Dict

# 第三方库
import aiohttp

# 本地模块（绝对）
from pyutagent.core.config import Settings

# 本地模块（相对）
from .base import BaseClass
```

#### RULE-014: 循环导入禁止
- **规则**: 禁止模块间循环导入
- **解决方案**:
  - 使用延迟导入
  - 提取公共接口到独立模块
  - 使用依赖注入

## 3. 测试规则

### 3.1 测试覆盖规则

#### RULE-015: 单元测试覆盖
- **规则**: 新代码单元测试覆盖率必须 > 90%
- **测量**: 使用 `pytest-cov`
- **报告**: 每次CI必须生成覆盖率报告

#### RULE-016: 集成测试覆盖
- **规则**: 所有公共API必须有集成测试
- **要求**:
  - 测试正常路径
  - 测试错误路径
  - 测试边界条件

#### RULE-017: 回归测试
- **规则**: 迁移后必须运行完整回归测试
- **范围**:
  - 单元测试
  - 集成测试
  - 端到端测试（如有）

### 3.2 测试质量规则

#### RULE-018: 测试独立性
- **规则**: 每个测试必须独立，不依赖其他测试
- **禁止**:
  - 共享状态
  - 测试顺序依赖
  - 全局状态修改

#### RULE-019: 测试命名
- **规则**: 测试函数名必须清晰描述测试内容
- **格式**: `test_<function>_<scenario>_<expected>`
- **示例**: `test_send_message_queue_full_raises_error`

#### RULE-020: 断言明确
- **规则**: 每个测试必须有明确的断言
- **要求**:
  - 断言失败信息清晰
  - 避免模糊断言（如 `assert result`）
  - 使用 `pytest.raises` 测试异常

## 4. 文档规则

### 4.1 代码文档规则

#### RULE-021: 公共API文档
- **规则**: 所有公共API必须有文档
- **内容**:
  - 功能描述
  - 使用示例
  - 参数说明
  - 返回值
  - 异常

#### RULE-022: 变更日志
- **规则**: 每个版本必须有变更日志
- **格式**: Keep a Changelog
- **内容**:
  - 新增功能
  - 变更
  - 废弃
  - 移除
  - 修复
  - 安全

### 4.2 架构文档规则

#### RULE-023: 架构决策记录
- **规则**: 重大架构决策必须有ADR
- **格式**:
```markdown
# ADR-XXX: 标题

## 状态
提议/已接受/已废弃

## 背景
问题描述

## 决策
决策内容

## 后果
正面和负面影响
```

#### RULE-024: 迁移文档
- **规则**: 每次迁移必须更新迁移文档
- **内容**:
  - 迁移步骤
  - 代码示例
  - 常见问题

## 5. 版本控制规则

### 5.1 提交规则

#### RULE-025: 提交信息规范
- **规则**: 提交信息必须遵循 Conventional Commits
- **格式**: `<type>(<scope>): <subject>`
- **类型**:
  - `feat`: 新功能
  - `fix`: Bug修复
  - `docs`: 文档
  - `style`: 代码格式
  - `refactor`: 重构
  - `test`: 测试
  - `chore`: 构建/工具

#### RULE-026: 提交粒度
- **规则**: 每个提交只做一件事
- **限制**:
  - 文件变更 < 10 个
  - 行变更 < 200 行
  - 功能单一

### 5.2 分支规则

#### RULE-027: 分支策略
- **规则**: 使用 Git Flow 或 GitHub Flow
- **要求**:
  - `main`: 稳定版本
  - `develop`: 开发版本
  - `feature/*`: 功能分支
  - `hotfix/*`: 紧急修复

#### RULE-028: 合并要求
- **规则**: 合并前必须通过所有检查
- **检查项**:
  - 代码审查
  - CI通过
  - 测试通过
  - 覆盖率达标

## 6. 性能规则

### 6.1 性能基准

#### RULE-029: 性能回归测试
- **规则**: 重大变更必须进行性能测试
- **基准**:
  - 消息总线: > 1000 msg/sec
  - Agent执行: < 100ms 延迟
  - 内存使用: 无泄漏

#### RULE-030: 性能优化记录
- **规则**: 性能优化必须有记录
- **内容**:
  - 优化前性能
  - 优化方法
  - 优化后性能
  - 测试方法

## 7. 安全规则

### 7.1 代码安全

#### RULE-031: 敏感信息保护
- **规则**: 禁止硬编码敏感信息
- **要求**:
  - 使用环境变量
  - 使用配置文件
  - 使用密钥管理服务

#### RULE-032: 输入验证
- **规则**: 所有外部输入必须验证
- **要求**:
  - 类型检查
  - 范围检查
  - 格式检查
  - 注入防护

### 7.2 依赖安全

#### RULE-033: 依赖扫描
- **规则**: 定期扫描依赖漏洞
- **工具**: `safety`, `snyk`, `dependabot`
- **频率**: 每周

#### RULE-034: 依赖更新
- **规则**: 及时更新有漏洞的依赖
- **要求**:
  - 高危漏洞: 24小时内
  - 中危漏洞: 1周内
  - 低危漏洞: 1月内

## 8. 合规检查清单

### 8.1 迁移前检查

- [ ] 所有测试通过
- [ ] 文档已更新

### 8.2 迁移中检查
- [ ] 小步提交
- [ ] 每次提交后测试通过
- [ ] 废弃警告正确
- [ ] 功能等价

### 8.3 迁移后检查
- [ ] 完整测试套件通过
- [ ] 迁移检查脚本通过

## 9. 违规处理

### 9.1 违规等级

| 等级 | 描述 | 处理 |
|------|------|------|
| 严重 | 导致功能回归或安全漏洞 | 立即修复，阻断发布 |
| 重要 | 影响代码质量或可维护性 | 限期修复，记录技术债 |
| 轻微 | 代码风格或文档问题 | 建议修复，不阻断 |

### 9.2 豁免流程
1. 提交豁免申请，说明原因
2. 技术负责人审批
3. 记录豁免决定
4. 定期回顾豁免
