# GUI 批量测试生成 - 编译策略选择指南

## 功能概述

PyUT Agent 的批量测试生成功能现在支持三种不同的编译策略，您可以根据项目需求选择最适合的模式。

## 编译策略说明

### 1. 标准模式 (Standard Mode) - 推荐用于高质量要求

**工作流程：**
```
文件 1: 生成 → 编译 → 测试 → 修复
文件 2: 生成 → 编译 → 测试 → 修复  
文件 3: 生成 → 编译 → 测试 → 修复
...
```

**特点：**
- ✅ 每个文件生成后立即验证
- ✅ 自动修复编译错误和测试失败
- ✅ 实时质量保障
- ⚠️ 生成速度较慢
- ⚠️ 每个文件都会进行完整的验证循环

**适用场景：**
- 关键业务代码测试
- 需要高质量保证的项目
- 复杂逻辑的测试生成
- 第一次使用项目时的测试生成

### 2. 延迟编译模式 (Defer Compilation) - 平衡速度与质量

**工作流程：**
```
阶段 1 - 生成所有文件:
  文件 1: 生成测试代码
  文件 2: 生成测试代码
  文件 3: 生成测试代码
  ...

阶段 2 - 统一编译检查:
  编译所有生成的测试文件
  报告编译错误
```

**特点：**
- ✅ 先生成所有测试代码
- ✅ 最后统一编译验证
- ✅ 比标准模式更快
- ⚠️ 错误反馈延迟
- ⚠️ 可能需要手动修复

**适用场景：**
- 中等规模批量生成（10-50 个文件）
- 快速原型测试
- 对已有代码的快速测试覆盖
- 时间较为紧迫的场景

### 3. 快速模式 (Fast Mode) - 最快速度

**工作流程：**
```
阶段 1 - 生成所有文件（无验证）:
  文件 1: 生成测试代码（跳过编译/测试）
  文件 2: 生成测试代码（跳过编译/测试）
  文件 3: 生成测试代码（跳过编译/测试）
  ...

阶段 2 - 最后编译一次:
  编译所有生成的测试文件
  报告编译错误
```

**特点：**
- ✅ 最快的生成速度
- ✅ 完全跳过生成过程中的验证
- ✅ 适合大规模快速生成
- ⚠️ 不自动修复错误
- ⚠️ 编译错误需要手动处理
- ⚠️ 质量无法保证

**适用场景：**
- 大规模批量生成（>50 个文件）
- 快速生成测试草稿
- 对测试代码质量要求不高
- 时间非常紧迫的场景

## 使用步骤

### 1. 打开批量生成对话框

**方法 1：** 菜单栏
```
Tools → Generate All Tests (Ctrl+Shift+G)
```

**方法 2：** 工具栏（如果有）
```
点击批量生成图标
```

### 2. 选择文件

在文件列表中：
- 使用 "Select All" 选择所有文件
- 使用 "Deselect All" 取消所有选择
- 手动勾选/取消特定文件

### 3. 配置参数

#### 基本配置
- **Parallel Workers**: 并行 worker 数量
  - 0 = 不限制
  - 1-20 = 具体数量
  - 建议：根据 CPU 核心数设置

- **Timeout (s)**: 每个文件的超时时间
  - 范围：60-3600 秒
  - 默认：300 秒（5 分钟）
  - 复杂类可适当增加

- **Coverage Target (%)**: 覆盖率目标
  - 范围：0-100%
  - 默认：80%
  - 建议：70-90%

- **Continue on error**: 错误时继续
  - ✓ 勾选：一个文件失败不影响其他文件
  - ☐ 不勾选：第一个错误后停止

#### 编译策略配置

在右侧的 "Compilation Strategy" 组中选择：

**Standard Mode** (标准模式)
```
[✓] Standard Mode
```
- 适合：高质量要求
- 特点：每个文件立即验证

**Defer Compilation** (延迟编译)
```
[✓] Defer Compilation  
```
- 适合：平衡速度与质量
- 特点：生成完成后统一编译

**Fast Mode** (快速模式)
```
[✓] Fast Mode (Compile Only at End)
```
- 适合：最快速度
- 特点：跳过所有中间验证

### 4. 开始生成

点击 "Start" 按钮开始批量生成。

生成过程中：
- 实时查看每个文件的状态
- 进度条显示总体进度
- 计时器显示已用时间
- 可以暂停/停止生成

### 5. 查看结果

生成完成后会显示：
- 成功/失败文件数量
- 编译结果（如果使用了延迟编译或快速模式）
- 总耗时
- 每个文件的详细信息

## 策略选择建议

### 项目规模建议

| 项目规模 | 文件数量 | 推荐模式 | 理由 |
|---------|---------|---------|------|
| 小型项目 | < 10 | Standard | 文件少，质量保证优先 |
| 中型项目 | 10-50 | Defer | 平衡速度与质量 |
| 大型项目 | 50-200 | Defer/Fast | 速度更重要 |
| 超大型项目 | > 200 | Fast | 快速生成，后续手动修复 |

### 代码类型建议

| 代码类型 | 推荐模式 | 理由 |
|---------|---------|------|
| 核心业务逻辑 | Standard | 质量至关重要 |
| 工具类 | Defer | 相对简单，可快速生成 |
| DAO/Repository | Fast | 模式固定，易于手动修复 |
| Controller | Defer | 需要验证但不过于复杂 |
| 遗留代码 | Standard | 可能有复杂依赖 |

### 时间紧迫度建议

| 时间情况 | 推荐模式 | 说明 |
|---------|---------|------|
| 时间充足 | Standard | 保证质量 |
| 时间一般 | Defer | 平衡选择 |
| 时间紧迫 | Fast | 快速生成 |
| 非常紧急 | Fast + 高并行 | 最大速度 |

## 常见问题

### Q1: 哪种模式最快？
**A:** Fast Mode > Defer Mode > Standard Mode

### Q2: 哪种模式质量最好？
**A:** Standard Mode（自动修复所有错误）

### Q3: 如果选择了 Fast Mode，编译错误怎么办？
**A:** 
1. 查看错误消息对话框
2. 手动导航到出错的测试文件
3. 根据编译错误信息修复代码
4. 重新运行 `mvn test-compile` 验证

### Q4: 可以中途切换模式吗？
**A:** 不可以。必须在开始生成前选择模式，生成过程中不能切换。

### Q5: Parallel Workers 设置多少合适？
**A:** 
- CPU 核心数少（4 核以下）：1-2
- CPU 核心数中等（4-8 核）：2-4
- CPU 核心数多（8 核以上）：4-8
- 内存有限：减少并行数

### Q6: 生成失败的文件可以重新生成吗？
**A:** 可以。关闭对话框后，重新打开批量生成对话框，只选择失败的文件再次生成。

## 最佳实践

### 1. 分批次生成
对于大型项目，建议分批生成：
```
第一批：核心模块 (Standard Mode)
第二批：业务模块 (Defer Mode)  
第三批：工具类 (Fast Mode)
```

### 2. 混合策略
```
重要代码：Standard Mode
一般代码：Defer Mode  
测试代码：Fast Mode
```

### 3. 渐进式生成
```
1. 先用 Standard Mode 生成少量文件
2. 评估生成质量
3. 调整参数
4. 大规模使用 Defer/Fast Mode
```

### 4. 定期保存
生成过程中定期保存项目，避免意外丢失。

### 5. 代码审查
生成的测试代码应该进行代码审查，确保：
- 测试断言合理
- 测试覆盖全面
- 代码风格一致

## 技术实现

### 关键组件

1. **BatchGenerateDialog** - 批量生成对话框
   - 提供 UI 界面
   - 编译策略选择
   - 进度显示

2. **BatchGenerateWorker** - 批量生成工作线程
   - 执行生成任务
   - 传递配置参数
   - 报告进度和结果

3. **BatchGenerator** - 批量生成器
   - 并行控制
   - 文件生成调度
   - 批量编译

### 配置传递流程

```
BatchGenerateDialog
  ↓ (用户选择策略)
BatchGenerateWorker  
  ↓ (defer_compilation, compile_only_at_end)
BatchConfig
  ↓ (配置参数)
BatchGenerator
  ↓ (执行生成)
ReActAgent
  ↓ (检查 skip_verification)
生成/验证测试代码
```
